## 📦 Deviations from spec

### 1. `npm create astro` was non-interactive-hostile on Windows PowerShell
**What:** The `--yes` flag did not skip every interactive prompt (deps install confirmation hung). Tried `echo "y" | npm create astro@latest -- ...` but only one `y` was consumed before EOF.

**How I worked around:** Created `package.json`, `astro.config.mjs`, `tsconfig.json`, `.gitignore` manually with the exact spec values, then ran `npm install` directly. Result is identical to what the wizard would have produced.

**Implication:** None — same final state.

### 2. No pre-existing `.htaccess` or `.cpanel.yml` in repo
**What:** The master prompt assumed `.htaccess` exists at repo root and would be moved to `public/`. Repo only had `index.html`, `.git/`, `.github/workflows/deploy.yml`.

**How I handled:** Created a fresh `public/.htaccess` with sensible defaults: HTTPS force, WordPress rewrite override (`RewriteRule ^ - [L]`), DirectoryIndex, cache headers (HTML 0s, assets 1y immutable), gzip compression, security headers (`X-Content-Type-Options`, `Referrer-Policy`, `Permissions-Policy`), `ErrorDocument 404 → /id/`, hidden-file blocking. **Review this file before going live** — confirm it matches what production currently serves.

`.cpanel.yml` was not present and not touched.

### 3. `base: '/id/'` does not put files in `dist/id/`
**What:** I noted in spec verification step "verify `dist/id/index.html` exists". Astro's `base` config does NOT physically nest output — it only prefixes URLs.

**How I handled:** Verified the actual structure works for the deploy target: files in `dist/` deploy to `/home/u1427052/public_html/id/`, which serves at `/id/...` matching the base config. URLs inside HTML are correctly prefixed with `/id/`.

### 4. Empty content collections cause warnings
**What:** Build emits 3 warnings: `[glob-loader] No files found matching "**/*.md" in directory "src/content/{paket,karya,journal}"`. These are non-blocking warnings.

**How I handled:** Left as-is. Will disappear once Wave 2 adds first content entries. Schema is valid and types are generated.

### 5. Lint warnings during dev (non-blocking)
**What:** TS lint shows `import.meta.env` warning in `src/utils/url.ts` and `astro/tsconfigs/strict not found` in `tsconfig.json` from the IDE.

**How I handled:** Both resolve at build time when Astro generates `.astro/types.d.ts`. Verified `npm run build` succeeds without errors.
